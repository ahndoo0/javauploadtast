개방적인 네트워크 
-인터넷은 세계적인 규모의 개방성으로 인해 방대한 정보의 바다라고 할 수 있으며 기존의 조직 내 정보시스템과의
통합이 용이하다. 즉 모뎀과 개인용  컴퓨터에 설치된 통신용 프로그램을 통해 접속하거나 전용회선을 통해
조직내에서 네트워크처럼 활용할 수 있다.

소유자, 운영자, 권력자가 없는 네트워크
-단지 네트워크에 연결된 각 컼ㅁ퓨터가 일정한 규칙에 의해 주소를 갖도록
유도하거나 정기적으로 점검하고 표준을 제시하는 등 일부 관리기관만 존재한다.

실시간 다방향의 멀팉미디어 네트워크
-기술의 발전으로 문자나 음성정보에 의해 동화상 정보의  전달이 가능하고 지역과 호스트 기종에 관계없이
상호간 데이터의 송수신이 가능하며 다양한 정보에 접근할 수 있다.

자체가 대중지향적인 네트워크
-모뎀이나 전화선과 같이 컴퓨터 통신을 위한 간단한 장치와 통신 소프트웨어만 있으면 시내 전화요금
수준으로 사용이 가능하고 검색 소프트웨어 기술의 발달로 영어해독만 가능하면 손쉽게 사용할  수 있다.

HTTP 프로토콜 특징
HTTP프로토콜을 무상태(stateless) 특성 때문에 이후 클라이언트가 다시 연결한다고 해도 서버는 이전의 연결에 대한 어떤 기억이나 결과도 가지고 있지 않는다.
- HTTP가 TCP의 서비스를 이용하지만, HTTP자체는 상태가 존재하지 않는 (stateless) 프로토콜이다. 즉 클라이언트 요청 메시지를 보내어 트랜잭션을 초기화
반면에 FTP와 같은 프로토콜은 연결이 닫히기 전에 많은 요청을 수행할 수 있따.
이와 같이 상태를 기억하지 않는 것을 HTTP의 약점처럼 보이지만 서버의 부하를최대로 줄일수 있다는 장점이 있다.

웹  프로그램의 이해
기본적으로 클라이언트(client)/서버(server) 방식의 형태
-클라이언트(웹브라우저)가 특정 웹서버에 요청(request)하게 되면 웹 서버가 이를 처리한 후 결과를 클라이언트에게 응답(response)을 하게 되는 구조
Client (browser)
-먼저 다양한 웹 브라우저를 통한 WWW클라이언트는 WWW서버에게 URL이라는 양식으로 원하는 서비스를 요청
Server
-서버는 WWW 클라이언트에게 서비스의 요청에 따른 정보를 HTML이라는 문서로 HTTP 프로토콜을 사용하여 전송

웹 프로그래밍 기술의 발전
-일반적으로 웹페이지를 정적 웹페이지와 동적 웹페이지로 분류

동적 웹페이지
-사용자(클라이언트)가 어떠한 요청을 하면 웹서버에서 요청 정보를 처리한 후 제작된 html문서를 사용자에게 보내준다.
-모든 사용자들은 요청에 따라 각각 다른 화면을 보게 된다.
-서버는 사용자에 맞는 html문서를 생성하여 사용자에게 응답하게 된다.
-즉, 모든 사용자는 서로 다른 결과를 서버로부터 응답 받게 된다.
- 동적 웹 페이지를 구현하기 위해서는 jsp & serviet, asp, asp.net, php, ruby, python 등의 웹프로그래밍 언어를 이용
ex) 어떠한 홈페이지 접속 시 매일 네이버에서 로그인 요청 시 로그인 결과화면은 회원마다 각각 다른 화면을 볼 수 있다. 
또한 권한별 웹페이지, 시간별 동적 웹페이지 등이 있다.

정적 웹페이지의 한계
-임의의 사용자가 페이지를 가져와서 브라웅저상에 보기 전에 이미 그 페이지의 모든 내용이 정해짐
-정적 콘텐츠는 사용자가 웹서버에 정해진 자원만 볼 수 있음
-사용자가 임의로 입력한 값을 서버에 반영하거나 반영된 값을 조회하는 등의 처리를 하기 위해서는 정적 콘텐츠로는 불가능

동적 웹페이지 
-사용자가 페이지를 요청하는 시점에 그 페이지를 새롭게 구성하는 행위를 수행
-정적 콘텐츠 방식과 달리 동적 콘텐츠는 사용자가 페이지를 요첯ㅇ하는 시점에 그 페이지를 새롭게 구성하는 행위를 수행
-동적 콘텐츠는 클라이엉ㄴ트 요청에 대한 응답으로 생성한 모든 종류의 데이터를 말함
-클라이언트와 상호 연등하기 위해 또는 사용자의 선택과 선호에 따라 응답 페이지를 생성할 때 사용

CGI vs  WAS
-웹 서버가 웹 애플리케이션을 실행 하는 방식에 따라 즉, 웹 서버가 직접 프로그램을 호출하는 여부 따라
-CGI방식 (실행코드 방식)
->웹 서버가 직접 애플리케이션을 호출하는구조
->[웹 브라우저 요청 전송]>[웹 서버 요청 받음]>[프로그램 실행]

-WAS방식 (스크립트 방식)
->웹 서버가 직접 프로그램을 호출하기 보다는 WAS를 통해서 간접적으로 애플리케이션 프로그램을 실행
->[웹 브라우저 요청 전송]>[웹 서버 요청 받음]>[스크립트 코드 번역]>[프로그램 실행]

-여러명의 사용자가 동일한 페이지를 요청하여 여러 애플리케이션 프로그램을 처리할 때 오직 한 개의 프로세스만을 할당하고 사용자의

-CGI에 비해 가지는 장점

->CGI방식에 비하여 CPU 점유도나 메모리 점유도에 있어서 상당히 효율적
->동시에 더 많은 웹브라우저의 요청을 처리할 수 있따는 것을 의미하며 이는 곧 안정적인 웹 서비스를 제공할 수 있는 기반
->각 웹 요청에 대해서 애플리케이션 서버는 스레드로서 처리

5.29
Summary:Class vs. ID
한 페이지 내에서 여러 번 반복될 필요가 있는 스타일은 class 선택자를 사용하고, 단 한번 유일하게 적용될 스타일은 ID 선택자를 
사용하는 것을 권장 
- class 속성은 어떤 분류 안에 포함된 요소의 특성을 정으의하는데 사용
-id 속성은 어떤 요소에 대해 유일한 특성을 정ㅇ의, 즉 HTML 문서에서 특정 id 속성값은 오직 하나만 있어야 함
class 선택자는 글자색이나 글자 굵기 등 나중에 다른 곳에도 적용할 수 있는 스타일을 지정하고 id 선택자는 웹 문서 안에서 
요소의 배치 방법을 지정할 때 자주 사용
class 속성은 속성값을 두 개 이상 가질 수 있어 class 선택자를 사용하면 한 태그 내에서도 여러 종류의 스타일 규칙을 적용할 수 있음.
ID 선택자의 우선순위가 class 선택자의 우선 순위 보다 높아 우선으로 적용되어야 할 스타일을 ID 선택자를 사용하여 
정의하는 것을 권장.
